# 공통 컨벤션 (전 레이어)

> 매 세션 항상 로드되는 전 레이어 공통 원칙입니다. 구현 디테일이 아니라 **잘 바뀌지 않는 큰 틀**만 간결하게 유지하세요.
> 특정 라이브러리·구현 패턴·프로젝트 고유 규칙은 레이어 파일(`02_*` ~ `05_*`)에 둡니다.

## 1. 아키텍처

### 1.1 레이어링 & 의존 방향

- 레이어를 명확히 분리하고, 의존은 한 방향(안쪽 코어를 향함)으로만 흐른다.
- 안쪽 레이어는 바깥쪽(프레임워크·I/O·전송 계층)을 알지 못한다.

### 1.2 경계 & 데이터 흐름

- 각 레이어는 자기 타입을 쓰고, 경계에서 명시적으로 변환한다. 내부 모델을 다른 레이어로 누출하지 않는다.

### 1.3 추상화

- 외부 시스템(DB·네트워크·메시징)은 인터페이스로 격리하고, 구체 구현은 가장 바깥 레이어에 둔다.

## 2. API / 계약

### 2.1 설계 규약

- 일관된 요청/응답 형태와 상태 표현을 따른다. 에러도 정해진 단일 포맷으로 응답한다.

### 2.2 버저닝 & 하위호환

- 공개 계약(API·이벤트·스키마) 변경은 하위호환을 유지하거나 명시적으로 버전을 올린다.

## 3. 데이터 / 영속성

### 3.1 트랜잭션

- 트랜잭션 경계를 명확히 정하고, 한 단위 작업은 원자적으로 처리한다.

### 3.2 스키마 변경 & 마이그레이션

- 스키마 변경은 버전 관리되는 마이그레이션으로만 한다. 수동 변경 금지.

### 3.3 일관성

- 데이터 정합성 규칙(제약·불변식)을 한 곳에서 강제한다.

## 4. 코드 일관성

### 4.1 네이밍

- 일관되고 예측 가능하게. 같은 종류의 것은 같은 규칙(접두/접미사 포함)을 따른다. (레이어별 상세 규칙은 각 레이어 파일)

### 4.2 구조

- 파일·모듈 배치를 예측 가능하게. "이건 어디 있을지"를 추측할 수 있어야 한다.

### 4.3 자동화

- 스타일은 포맷터·린터로 기계가 강제한다. 코드 리뷰에서 스타일을 논쟁하지 않는다.

## 5. 에러 처리

### 5.1 표현

- 라이브러리·도메인 경계의 에러는 타입으로 명확히 표현한다. 문자열 에러를 지양한다.

### 5.2 전파 & 변환

- 에러를 삼키지 않고 전파한다. 레이어 경계에서 그 레이어의 에러 타입으로 변환한다.

### 5.3 실패 정책

- 복구 불가능한 상태에서만 중단한다. 라이브러리 코드에서 임의 중단(panic 등)은 금지.

## 6. 관측성

### 6.1 로깅

- 사람이 읽는 문자열이 아니라 구조적 필드로 로깅한다.
- 로그 레벨의 의미를 고정한다: ERROR=처리 실패·조치 필요, WARN=비정상이나 계속 가능, INFO=주요 상태 전이, DEBUG=진단용. (프로덕션 기본 INFO)
- 민감정보(시크릿·토큰·PII)는 로깅하지 않는다. 불가피하면 마스킹한다.
- 같은 사건은 한 경계에서 한 번만 로깅한다. 에러를 로깅하면서 다시 던지지 않는다(log-and-throw 금지 — 5.2 참고).

### 6.2 메트릭

- 핵심 지표(처리량·지연·에러율)를 계측한다.

### 6.3 트레이싱

- 요청을 가로지르는 상관관계 식별자(trace / correlation id)를 전파한다.

### 6.4 헬스체크

- liveness / readiness를 구분해 노출한다.

## 7. 동시성 / 복원력

### 7.1 논블로킹

- 비동기 런타임 / 이벤트 루프를 블로킹하지 않는다.

### 7.2 타임아웃

- 모든 외부 호출에는 타임아웃을 둔다. 무한정 대기하지 않는다.

### 7.3 재시도 & 멱등성

- 재시도는 백오프와 함께. 재시도 가능한 연산은 멱등하게 설계한다.

### 7.4 Graceful shutdown

- 종료 신호에 진행 중 작업을 정리하고 연결을 드레인한 뒤 종료한다.

## 8. 보안

### 8.1 입력 / 신뢰 경계

- 외부 입력은 진입점에서 검증한다. 신뢰 경계를 명확히 한다.

### 8.2 인증 & 인가

- 인증과 인가를 분리하고, 기본은 거부(deny by default)로 둔다.

### 8.3 시크릿 & 데이터 보호

- 비밀값은 코드에 두지 않고 주입한다. 민감 데이터는 전송·저장 시 보호한다.

## 9. 설정

### 9.1 환경 분리

- 환경별 설정을 코드와 분리하고 환경 변수 / 설정 소스로 주입한다.

### 9.2 시작 시 검증

- 필수 설정은 부팅 시 검증하고, 누락 시 빠르게 실패한다.

## 10. 테스트

### 10.1 피라미드

- 단위 테스트를 기본으로, 통합 / E2E는 핵심 흐름에 집중한다.

### 10.2 격리 & 결정성

- 테스트는 서로 독립적이고 결정적이어야 한다. 외부 의존은 대체하거나 격리한다.

## 11. 의존성

### 11.1 버전 관리

- 의존성 추가는 신중히. 버전은 한 곳에서 단일하게 관리한다.

### 11.2 공급망 / 취약점

- 의존성 취약점을 주기적으로 점검한다.

## 12. 버전 관리

### 12.1 커밋

- 일관된 커밋 컨벤션(Conventional Commits 등)을 따른다.

### 12.2 브랜치 / PR

- 짧게 살아있는 브랜치와 작은 PR을 선호한다.
