# Willog Issue Shaping (Asana)

> **[필수 규칙]** 이 스킬은 반드시 `AskUserQuestion` 도구로 사용자의 최종 확인을 받은 후에만 Asana 태스크를 생성한다. 확인 없이 태스크를 만드는 것은 금지된다.

윌로그는 회사 업무를 Asana로 관리한다. 이 스킬은 개발 작업을 바로 만들지 않고, 먼저 양식(template)에 맞춰 내용을 함께 다듬은 뒤(shaping) 사용자 승인을 받아 Asana 태스크로 생성한다. 양식을 강제하는 이유는, 배경·요구사항·완료 조건이 빠진 채로 만들어진 태스크가 나중에 누가 봐도 맥락을 알 수 없게 되는 것을 막기 위함이다.

GitHub을 쓰는 개인 프로젝트용 `issue-shaping` 스킬과 역할이 같지만 대상이 Asana라는 점이 다르다.

## 사전 조건

- Asana MCP(`claude_ai_Asana`)가 연결되어 있어야 한다. 연결 여부는 워크플로우 1단계에서 `get_me`로 확인한다.
- 양식 파일은 이 스킬 폴더의 `issue-template/*.md`에 있다. 새 타입을 추가하려면 파일을 추가하면 된다.

## 양식(template) 파일 형식

`issue-template/` 하위 각 `.md`는 YAML frontmatter + 마크다운 body로 구성된다.

```yaml
---
name: Feature              # 사용자에게 보여줄 양식 이름
about: 새 기능 개발         # 한 줄 설명 (어떤 상황에 쓰는지)
type_token: "[Feat]"       # 태스크명 앞에 붙는 타입 토큰
custom_fields:             # (선택) 필드명 → 옵션명 힌트. GID 아님, 텍스트.
  TaskType: "✨ Feature"
---
## 배경 (Why)
...
```

**핵심 설계 — 커스텀필드는 실행 시점에 대상 프로젝트에서 동적으로 결정한다.** 프로젝트마다 커스텀필드의 종류·이름·옵션·GID가 완전히 다르다. 예를 들어 `[BE][Platform] MetricStore`에는 양식이 가정하는 `TaskType` 필드가 **없고**, 대신 `우선순위`·`상태`·`진행 단계`·`QA 필요 여부` 같은 필드가 있다. 그래서 양식의 `custom_fields`는 **고정값이 아니라 힌트**로만 쓴다: 대상 프로젝트에 같은 이름의 필드·옵션이 있으면 추천 기본값으로 자동 선택하고, 없으면 무시한다. 실제로 채울 필드는 4.3에서 대상 프로젝트의 필드 정의를 조회해 `AskUserQuestion`으로 사용자에게 직접 고르게 한다. 이렇게 해야 양식 하나로 필드 체계가 서로 다른 모든 프로젝트에 대응할 수 있다.

## 워크플로우

### 1단계: 연결 검증

`get_me`를 호출한다. 실패하면(인증 안 됨 등) "Asana MCP가 연결되어 있지 않습니다. 연결 후 다시 시도하세요."를 출력하고 **종료**한다. 성공하면 응답의 사용자 GID와 workspace를 이후 단계에서 재사용한다.

### 2단계: 대상 프로젝트·섹션 결정

#### 2.1 프로젝트 결정

사용자가 메시지에서 프로젝트를 명시했는지 확인한다(예: "DMS에 만들어줘", "[BE][Solution]Core에").

- **명시됨**: `search_objects`(`resource_type: "project"`, `query: <프로젝트명>`)로 검색한다.
  - 정확히 1개 매칭 → 그 프로젝트 사용.
  - 여러 개 매칭 → `AskUserQuestion`으로 어느 프로젝트인지 고르게 한다.
  - 0개 → "해당 이름의 프로젝트를 찾지 못했습니다."를 알리고, 아래 '명시 안 됨' 흐름으로 넘어간다.
- **명시 안 됨**: `get_teams`로 사용자가 속한 팀을 조회하고, 각 팀에 대해 `get_projects`(`team` 필터)로 프로젝트 목록을 모은다. 그 목록을 `AskUserQuestion`으로 보여주고 선택받는다. 프로젝트가 4개를 넘으면 상위 항목 + "직접 입력" 옵션을 제공하고, 직접 입력 시 전체 목록을 텍스트로 나열한 뒤 고르게 한다.

#### 2.2 섹션 결정

선택된 프로젝트를 `get_project`(`include_sections: true`, `opt_fields: "custom_field_settings.custom_field.name,custom_field_settings.custom_field.enum_options.name"`)로 조회한다. 이 한 번의 호출로 **섹션 목록**과 **커스텀필드 정의**를 동시에 확보한다(4단계에서 재사용).

섹션이 여러 개면 `AskUserQuestion`으로 어느 섹션에 넣을지 고르게 한다. "백로그"나 "할일"처럼 명백한 시작 섹션이 있으면 그것을 첫 번째(추천) 옵션으로 둔다.

### 3단계: 양식 선택

#### 3.1 양식 목록화·파싱

`Glob`으로 이 스킬 폴더의 `issue-template/*.md`를 조회하고, 각 파일의 frontmatter(`name`, `about`, `type_token`, `custom_fields`)와 body를 파싱한다. 파싱 실패한 파일은 제외한다.

#### 3.2 한 줄 설명 수집

최초 사용자 메시지에 작업 요약이 이미 있으면(예: "로그인 세션 만료 버그") 그대로 사용한다. 없으면 한 번 묻는다(일반 텍스트 질문): "어떤 작업을 shaping할까요? 한 줄로 요약해주세요."

#### 3.3 양식 결정

- 양식이 1개면 질문 없이 사용한다.
- 여러 개면 한 줄 설명을 근거로 가장 맞는 양식을 추론하고(name + about 비교), `AskUserQuestion`으로 확인한다. "다른 양식 선택"을 고르면 전체 양식 목록에서 다시 고르게 한다.

### 4단계: Draft 작성 및 Review

#### 4.1 태스크명 생성

`태스크명 = [컴포넌트]<type_token> 요약` 형태로 만든다.

- `type_token`은 선택된 양식에서 가져온다(예: `[Feat]`).
- 컴포넌트(`[BE]`/`[FE]` 등)는 한 줄 설명에서 추론해 앞에 붙인다. 백엔드 작업으로 보이면 `[BE]`, 프론트면 `[FE]`, 불확실하면 생략한다. Review 단계에서 사용자가 고칠 수 있으므로 과하게 고민하지 않는다.
- 요약은 명사구, 50자 이하 목표. 예: `[BE][Fix] 로그인 후 세션 만료 시 무한 redirect`

#### 4.2 본문(notes) 생성

선택된 양식의 body를 복사하고, 각 `## 섹션` 아래를 한 줄 설명과 이전 대화 맥락으로 채운다. 정보가 부족한 섹션은 placeholder를 남기지 말고 추론 가능한 만큼 최선의 추측으로 채운다(Review에서 사용자가 다듬는다).

Asana에 넣을 때는 `html_notes`로 변환한다. Asana가 허용하는 태그는 `<body><h1><h2><strong><em><u><s><code><ol><ul><li><a><blockquote><pre>` 뿐이다. `<h3>`·`<hr>`·`<table>`·체크박스는 렌더링되지 않으므로 쓰지 않는다.

**변환 스타일 — 미니멀(이모지·구분선 없음).** 섹션 구분은 `<h2>`의 여백이 담당하므로 별도 구분선을 넣지 않는다. 아래 규칙을 그대로 적용해 결과가 매번 동일하게 나오도록 한다(곤란해서 평문으로 폴백하지 말 것 — 규칙대로 변환하면 항상 well-formed가 된다).

| 마크다운 | html_notes |
|---|---|
| `## 제목` | `<h2>제목</h2>` |
| 일반 단락 | 텍스트 그대로 (h2/리스트 사이에 둠) |
| `- 항목` / `* 항목` | `<ul><li>항목</li></ul>` (연속 항목은 하나의 `<ul>`로 묶음) |
| `- [ ] 항목` / `- [x] 항목` | 체크박스 마커(`[ ]`·`[x]`)는 **제거**하고 `<li>항목</li>`로 (Asana는 본문 체크박스 미지원) |
| `1. 항목` | `<ol><li>항목</li></ol>` (연속 항목은 하나의 `<ol>`로 묶음) |
| `라벨: 값` 형태의 메타 줄 (예: `환경: Prod`, `디자인 / 스펙: ...`) | `<strong>라벨:</strong> 값` — 라벨만 굵게 |
| `**강조**` | `<strong>강조</strong>` |
| `` `코드` `` | `<code>코드</code>` |
| `[텍스트](url)` | `<a href="url">텍스트</a>` |
| 코드블록(``` ```` ```) | `<pre>...</pre>` |

추가 규칙:
- `&` `<` `>`는 각각 `&amp;` `&lt;` `&gt;`로 이스케이프한다(코드/URL 포함).
- 양식의 안내문(예: "이 기능이 해결하는 문제 / 필요한 이유")은 그대로 넣지 말고 4.2 도입부대로 실제 내용으로 채운다. 내용이 채워진 섹션만 남기고, 정보가 전혀 없어 추론도 불가한 섹션은 헤더째 생략한다.
- 전체를 단일 `<body>...</body>`로 감싼 well-formed XML로 만든다.

#### 4.3 커스텀필드 동적 채우기

2.2에서 받은 대상 프로젝트의 `custom_field_settings`를 사용한다(추가 호출 없음). 양식에 박힌 고정 필드를 그대로 쓰지 않고, **대상 프로젝트에 실제로 존재하는 필드**를 기준으로 값을 정한다.

1. **후보 추리기.** `custom_field_settings`에서 `enum`/`multi_enum` 타입 필드만 후보로 삼는다. `text`/`number`/`date`/`people` 타입은 본문·맥락에서 명확히 추론될 때만 채우고, 아니면 비워 둔다. 후보를 중요도 순으로 정렬한다: 우선순위류 → 상태/진행 단계류 → QA류 → 나머지.

2. **추천값 추론.** 각 후보 필드마다 추천 옵션 하나를 고른다.
   - 양식의 `custom_fields` 힌트에 같은 **필드명+옵션명**이 있으면 그 옵션을 추천값으로 삼는다.
   - 상태/단계 계열은 2.2에서 고른 시작 섹션에 대응하는 옵션(예: 섹션이 "백로그"면 상태 "진행 전")을 추천한다.
   - 우선순위 계열은 본문의 긴급도 표현에서 추론하고, 불확실하면 중간값(예: "P2 보통")을 추천한다.

3. **사용자 확인 (`AskUserQuestion`).** 후보 중 상위 최대 4개 필드를 한 번의 `AskUserQuestion`(질문 4개 한도)으로 묶어 묻는다. 각 질문 = 필드 1개:
   - 질문 `header`는 필드명, options는 **추천 옵션을 맨 앞**(라벨 끝에 "(추천)")에 두고 자주 쓰는 옵션 몇 개 + `"설정 안 함"`을 둔다.
   - 옵션이 4개를 넘는 필드(예: `상태`·`진행 단계`)는 전부 못 넣으므로, 그 외 값은 사용자가 "기타"(직접 입력)로 정확한 옵션명을 적게 한다. 직접 입력값은 그 필드의 `enum_options[].name`과 대조하고, 매칭 실패 시 그 필드는 설정 안 함 처리 + 경고 한 줄.
   - 후보가 4개를 넘으면, 5번째 이후 필드는 추천값을 그대로 적용하되 4.4 draft에 "(자동)"으로 표시해 Review에서 고칠 수 있게 한다.
   - 매칭되는 enum 후보 필드가 하나도 없으면 이 질문을 생략한다.

4. **GID 변환.** 확정된 `필드명→옵션명`을 `custom_field_settings`로 GID 변환해 `{필드GID: 옵션GID, ...}`를 만든다. enum이면 옵션명으로 `enum_options[].name`을 찾아 옵션 GID를 얻는다.

#### 4.4 Draft 출력

채팅창에 전체 draft를 inline으로 보여준다:

```
프로젝트: <프로젝트명> / 섹션: <섹션명>
태스크명: <생성된 태스크명>
담당자: 나(me)
커스텀필드: <필드명=옵션명, ... (자동 적용분은 "(자동)" 표시, 없으면 "없음")>
---
<본문 전체>
---
```

#### 4.5 Review 루프

사용자가 수정 요청을 하면 반영해 draft 전체를 다시 출력한다. 다음까지 반복한다:

- 수락 의사("좋아", "OK", "진행해") → 5단계로
- 중단 의사("취소", "그만") → 조용히 종료

### 5단계: 최종 승인 [절대 건너뛰지 않는다]

`AskUserQuestion`으로 최종 확인을 받는다.

- **question**: `"위 draft로 Asana 태스크를 생성할까요?"`
- **header**: `"태스크 생성 확인"`
- **옵션 1**: `"태스크 생성"` → 6단계로
- **옵션 2**: `"취소"` → 조용히 종료

### 6단계: 태스크 생성

`create_tasks`를 호출한다. 태스크 1개를 만든다:

- `project_id`: 2.1에서 정한 프로젝트 GID
- `section_id`: 2.2에서 정한 섹션 GID
- `name`: 4.1의 태스크명
- `html_notes`(또는 `notes`): 4.2의 본문
- `assignee`: `"me"`
- `custom_fields`: 4.3의 매핑 결과를 JSON 문자열로 전달(`create_tasks`의 태스크별 `custom_fields`는 문자열 타입이다). 비어 있으면 생략.

**결과 처리:**
- 성공 → 생성된 태스크의 `permalink_url`을 `"태스크 생성됨: <url>"` 형식으로 출력하고 종료.
- 실패 → 반환된 오류 메시지를 그대로 보여주고, draft 본문은 이미 화면에 있으므로 다시 시도할 수 있게 안내한다.

## 오류 처리 요약

| 단계 | 상황 | 처리 |
|---|---|---|
| 1 | Asana MCP 미연결 | 안내 후 종료 |
| 2 | 명시한 프로젝트 못 찾음 | 알리고 팀 프로젝트 목록 선택으로 전환 |
| 2 | 팀/프로젝트 조회 실패 | 오류 안내 후 종료 |
| 3 | 양식 0개 / 전체 파싱 실패 | "issue-template에 양식이 없습니다" 안내 후 종료 |
| 4 | enum 후보 필드 없음 | 4.3 질문 생략, 커스텀필드 없이 진행 |
| 4 | 직접 입력 옵션명이 enum과 불일치 | 해당 필드 설정 안 함 + 경고 한 줄 |
| 5 | 사용자가 "취소" | 조용히 종료 |
| 6 | `create_tasks` 실패 | 오류 메시지 출력, 재시도 안내 |
