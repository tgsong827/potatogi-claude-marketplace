# Willog Issue Shaping (Asana)

> **[필수 규칙]** 이 스킬은 반드시 `AskUserQuestion` 도구로 사용자의 최종 확인을 받은 후에만 Asana 태스크를 생성한다. 확인 없이 태스크를 만드는 것은 금지된다.

윌로그는 회사 업무를 Asana로 관리한다. 이 스킬은 개발 작업을 바로 만들지 않고, 먼저 양식(template)에 맞춰 내용을 함께 다듬은 뒤(shaping) 사용자 승인을 받아 Asana 태스크로 생성한다. 양식을 강제하는 이유는, 배경·요구사항·완료 조건이 빠진 채로 만들어진 태스크가 나중에 누가 봐도 맥락을 알 수 없게 되는 것을 막기 위함이다.

GitHub을 쓰는 개인 프로젝트용 `issue-shaping` 스킬과 역할이 같지만 대상이 Asana라는 점이 다르다.

## 이슈 관리 워크플로우 (중요)

개발 태스크는 허브 프로젝트 `[SW][Platform] Metaboard`를 경유해 만들 수 있다:

1. `[SW][Platform] Metaboard`에 태스크를 **먼저 생성**한다.
2. 생성된 태스크를 **개별 관리 프로젝트에 추가로 연결**한다(`update_tasks`의 `add_projects`).

Asana에서 하나의 태스크는 여러 프로젝트에 동시에 속할 수 있고, 각 프로젝트 안에서 서로 다른 섹션·커스텀필드를 가진다. 그래서 Metaboard 경유 시에는 **두 프로젝트 모두**의 섹션·필드를 채운다.

단, 모든 태스크가 Metaboard를 거치는 것은 아니므로 **경유 여부는 태스크마다 사용자에게 확인**한다(3단계). 경유하지 않으면 개별 프로젝트에만 생성한다.

## 사전 조건

- Asana MCP(`claude_ai_Asana`)가 연결되어 있어야 한다. 연결 여부는 워크플로우 1단계에서 `get_me`로 확인한다.
- 양식 파일은 이 스킬 폴더의 `issue-template/*.md`에 있다. 새 타입을 추가하려면 파일을 추가하면 된다.

## 양식(template) 파일 형식

`issue-template/` 하위 각 `.md`는 YAML frontmatter + 마크다운 body로 구성된다.

```yaml
---
name: Feature              # 사용자에게 보여줄 양식 이름
about: 새 기능 개발         # 한 줄 설명 (어떤 상황에 쓰는지)
type_token: "[Feat]"       # 태스크명 앞에 붙는 타입 토큰
custom_fields:             # (선택) 필드명 → 옵션명 힌트. GID 아님, 텍스트.
  TaskType: "✨ Feature"
---
## 배경 (Why)
...
```

**핵심 설계 — 커스텀필드는 실행 시점에 대상 프로젝트에서 동적으로 결정한다.** 프로젝트마다 커스텀필드의 종류·이름·옵션·GID가 완전히 다르다. 예를 들어 `[BE][Platform] MetricStore`에는 양식이 가정하는 `TaskType` 필드가 **없고**, 대신 `우선순위`·`상태`·`진행 단계`·`QA 필요 여부` 같은 필드가 있다. 그래서 양식의 `custom_fields`는 **고정값이 아니라 힌트**로만 쓴다: 대상 프로젝트에 같은 이름의 필드·옵션이 있으면 추천 기본값으로 자동 선택하고, 없으면 무시한다. 실제로 채울 필드는 6.3에서 대상 프로젝트(들)의 필드 정의를 조회해 `AskUserQuestion`으로 사용자에게 직접 고르게 한다. 이렇게 해야 양식 하나로 필드 체계가 서로 다른 모든 프로젝트에 대응할 수 있다.

## 워크플로우

### 1단계: 연결 검증

`get_me`를 호출한다. 실패하면(인증 안 됨 등) "Asana MCP가 연결되어 있지 않습니다. 연결 후 다시 시도하세요."를 출력하고 **종료**한다. 성공하면 응답의 사용자 GID와 workspace를 이후 단계에서 재사용한다.

### 2단계: 개별 관리 프로젝트 결정

태스크가 속할 **개별 관리 프로젝트**를 먼저 정한다(Metaboard 경유 여부와 무관하게 항상 필요하다). 사용자가 메시지에서 프로젝트를 명시했는지 확인한다(예: "DMS에 만들어줘", "[BE][Solution]Core에").

- **명시됨**: `search_objects`(`resource_type: "project"`, `query: <프로젝트명>`)로 검색한다.
  - 정확히 1개 매칭 → 그 프로젝트 사용.
  - 여러 개 매칭 → `AskUserQuestion`으로 어느 프로젝트인지 고르게 한다.
  - 0개 → "해당 이름의 프로젝트를 찾지 못했습니다."를 알리고, 아래 '명시 안 됨' 흐름으로 넘어간다.
- **명시 안 됨**: `get_teams`(`user`: 1단계의 사용자 GID)로 사용자가 속한 팀을 조회하고(이 필터를 빼면 워크스페이스 전체 팀이 나온다), 각 팀에 대해 `get_projects`(`team`: 팀 GID)로 프로젝트 목록을 모은다. 그 목록을 `AskUserQuestion`으로 보여주고 선택받는다. 프로젝트가 4개를 넘으면 상위 항목 + "직접 입력" 옵션을 제공하고, 직접 입력 시 전체 목록을 텍스트로 나열한 뒤 고르게 한다.

### 3단계: Metaboard 경유 여부 결정

`AskUserQuestion`으로 이 태스크를 허브 프로젝트 `[SW][Platform] Metaboard`에 먼저 만들지 묻는다. 모든 태스크가 경유하는 것은 아니므로 매번 확인한다.

- **question**: `"이 태스크를 [SW][Platform] Metaboard에 먼저 생성하고 <개별프로젝트명>에 연결할까요?"`
- **header**: `"생성 방식"`
- **옵션 1**: `"Metaboard 경유 후 연결 (추천)"` → 경유 흐름
- **옵션 2**: `"<개별프로젝트명>에만 생성"` → 개별 흐름

**경유**를 고르면 Metaboard 프로젝트 GID가 필요하다. `search_objects`(`resource_type: "project"`, `query: "Metaboard"`)로 찾는다. 반환 결과 중 이름이 정확히 `[SW][Platform] Metaboard`인 항목을 쓴다. 이름이 정확히 일치하는 게 없고 후보가 여러 개면 `AskUserQuestion`으로 고르게 하고, 0개면 "Metaboard 프로젝트를 찾지 못했습니다. 개별 프로젝트에만 생성합니다."를 알리고 개별 흐름으로 전환한다.

이 선택의 결과로 **대상 프로젝트 목록**이 정해진다: 경유면 `[Metaboard, 개별프로젝트]`, 아니면 `[개별프로젝트]`. 이후 단계는 이 목록을 기준으로 움직인다.

### 4단계: 대상 프로젝트들의 섹션·필드 정의 수집

3단계에서 정해진 **대상 프로젝트 각각**을 `get_project`로 조회한다. 한 프로젝트당 한 번의 호출로 그 프로젝트의 **섹션 목록**과 **커스텀필드 정의**를 동시에 확보한다(6단계에서 재사용). 경유 흐름이면 Metaboard와 개별 프로젝트 각각에 대해 호출한다.

- `include_sections: true` — 섹션 목록(GID·이름)을 포함시킨다.
- `opt_fields`: `"custom_field_settings.custom_field.gid,custom_field_settings.custom_field.name,custom_field_settings.custom_field.resource_subtype,custom_field_settings.custom_field.enum_options.gid,custom_field_settings.custom_field.enum_options.name,custom_field_settings.custom_field.enum_options.enabled"`

`resource_subtype`(필드 타입: `enum`/`multi_enum`/`text`/`number`/`date`/`people`)는 6.3에서 enum 후보를 거르는 기준이므로 반드시 받아야 한다. 필드·옵션의 `gid`는 6.5에서 값을 `{필드GID: 옵션GID}`로 변환하는 데 쓰고, `enum_options.enabled`로 비활성 옵션은 추천·후보에서 제외한다.

섹션은 6.3에서 다른 필드들과 함께 묶어 결정하므로 여기서는 조회만 한다.

### 5단계: 양식 선택

#### 5.1 양식 목록화·파싱

`Glob`으로 이 스킬 폴더의 `issue-template/*.md`를 조회하고, 각 파일의 frontmatter(`name`, `about`, `type_token`, `custom_fields`)와 body를 파싱한다. 파싱 실패한 파일은 제외한다.

#### 5.2 한 줄 설명 수집

최초 사용자 메시지에 작업 요약이 이미 있으면(예: "로그인 세션 만료 버그") 그대로 사용한다. 없으면 한 번 묻는다(일반 텍스트 질문): "어떤 작업을 shaping할까요? 한 줄로 요약해주세요."

#### 5.3 양식 결정

- 양식이 1개면 질문 없이 사용한다.
- 여러 개면 한 줄 설명을 근거로 가장 맞는 양식을 추론하고(name + about 비교), `AskUserQuestion`으로 확인한다. "다른 양식 선택"을 고르면 전체 양식 목록에서 다시 고르게 한다.

### 6단계: Draft 작성 및 Review

#### 6.1 태스크명 생성

`태스크명 = [컴포넌트]<type_token> 요약` 형태로 만든다.

- `type_token`은 선택된 양식에서 가져온다(예: `[Feat]`).
- 컴포넌트(`[BE]`/`[FE]` 등)는 한 줄 설명에서 추론해 앞에 붙인다. 백엔드 작업으로 보이면 `[BE]`, 프론트면 `[FE]`, 불확실하면 생략한다. Review 단계에서 사용자가 고칠 수 있으므로 과하게 고민하지 않는다.
- 요약은 명사구, 50자 이하 목표. 예: `[BE][Fix] 로그인 후 세션 만료 시 무한 redirect`

#### 6.2 본문(notes) 생성

선택된 양식의 body를 복사하고, 각 `## 섹션` 아래를 한 줄 설명과 이전 대화 맥락으로 채운다. 정보가 부족한 섹션은 placeholder를 남기지 말고 추론 가능한 만큼 최선의 추측으로 채운다(Review에서 사용자가 다듬는다).

Asana에 넣을 때는 `html_notes`로 변환한다. Asana가 허용하는 태그는 `<body><h1><h2><strong><em><u><s><code><ol><ul><li><a><blockquote><pre>` 뿐이다. `<h3>`·`<hr>`·`<table>`·체크박스는 렌더링되지 않으므로 쓰지 않는다.

**변환 스타일 — 미니멀(이모지·구분선 없음).** 섹션 구분은 `<h2>`의 여백이 담당하므로 별도 구분선을 넣지 않는다. 아래 규칙을 그대로 적용해 결과가 매번 동일하게 나오도록 한다(곤란해서 평문으로 폴백하지 말 것 — 규칙대로 변환하면 항상 well-formed가 된다).

| 마크다운 | html_notes |
|---|---|
| `## 제목` | `<h2>제목</h2>` |
| 일반 단락 | 텍스트 그대로 (h2/리스트 사이에 둠) |
| `- 항목` / `* 항목` | `<ul><li>항목</li></ul>` (연속 항목은 하나의 `<ul>`로 묶음) |
| `- [ ] 항목` / `- [x] 항목` | 체크박스 마커(`[ ]`·`[x]`)는 **제거**하고 `<li>항목</li>`로 (Asana는 본문 체크박스 미지원) |
| `1. 항목` | `<ol><li>항목</li></ol>` (연속 항목은 하나의 `<ol>`로 묶음) |
| `라벨: 값` 형태의 메타 줄 (예: `환경: Prod`, `디자인 / 스펙: ...`) | `<strong>라벨:</strong> 값` — 라벨만 굵게 |
| `**강조**` | `<strong>강조</strong>` |
| `` `코드` `` | `<code>코드</code>` |
| `[텍스트](url)` | `<a href="url">텍스트</a>` |
| 코드블록(``` ```` ```) | `<pre>...</pre>` |

추가 규칙:
- `&` `<` `>`는 각각 `&amp;` `&lt;` `&gt;`로 이스케이프한다(코드/URL 포함).
- 양식의 안내문(예: "이 기능이 해결하는 문제 / 필요한 이유")은 그대로 넣지 말고 6.2 도입부대로 실제 내용으로 채운다. 내용이 채워진 섹션만 남기고, 정보가 전혀 없어 추론도 불가한 섹션은 헤더째 생략한다.
- 전체를 단일 `<body>...</body>`로 감싼 well-formed XML로 만든다.

#### 6.3 섹션·커스텀필드 통합 결정

태스크가 속할 **각 대상 프로젝트**(경유면 Metaboard + 개별, 아니면 개별 하나)는 자기 섹션과 커스텀필드를 따로 가진다. 이들을 **하나의 결정 묶음**으로 모아 `AskUserQuestion` 한 번으로 처리한다. 프로젝트마다 따로 묻지 않는 이유는, 질문이 두세 번으로 쪼개지면 사용자 흐름이 끊기기 때문이다.

1. **결정 항목 모으기.** 4단계에서 조회한 각 프로젝트 정의에서 다음을 항목으로 만든다.
   - **섹션 항목**: 프로젝트당 1개(섹션이 2개 이상일 때만). `header`는 프로젝트를 구분할 수 있게 짓는다(예: `"Metaboard 섹션"`, `"<개별프로젝트> 섹션"`).
   - **커스텀필드 항목**: 각 프로젝트의 `custom_field_settings`에서 `enum`/`multi_enum` 타입만. 두 프로젝트에 **같은 GID의 필드**가 겹치면 하나로 합친다(한 번 정하면 양쪽에 같은 값이 적용된다). `text`/`number`/`date`/`people` 타입은 본문·맥락에서 명확할 때만 채우고 아니면 비워 둔다.

2. **우선순위 정렬.** 항목을 중요도 순으로 정렬한다: 섹션 → 우선순위류 → 상태/진행 단계류 → QA류 → 나머지. 섹션을 앞에 두는 이유는 태스크가 보드 어디에 놓이는지가 가장 눈에 띄는 결정이기 때문이다.

3. **추천값 추론.** 각 항목마다 추천 하나를 고른다.
   - 섹션: "백로그"·"할일"처럼 명백한 시작 섹션을 추천.
   - 커스텀필드: 양식의 `custom_fields` 힌트에 같은 **필드명+옵션명**이 있으면 그 옵션. 상태/단계 계열은 고른 시작 섹션에 대응하는 옵션(섹션이 "백로그"면 상태 "진행 전"). 우선순위 계열은 본문 긴급도에서 추론하고 불확실하면 중간값(예: "P2 보통").

4. **사용자 확인 (`AskUserQuestion`).** 정렬된 항목 중 **상위 최대 4개**를 한 번의 질문(질문 4개 한도)으로 묶어 묻는다. 각 질문 = 항목 1개:
   - `header`는 항목명(섹션이면 프로젝트 구분 포함, 필드면 필드명), options는 **추천을 맨 앞**(라벨 끝에 "(추천)")에 두고 자주 쓰는 옵션 몇 개 + `"설정 안 함"`을 둔다.
   - 옵션이 4개를 넘는 항목(예: `상태`·`진행 단계`)은 전부 못 넣으므로, 그 외 값은 "기타"(직접 입력)로 정확한 이름을 적게 한다. 직접 입력값은 해당 항목의 후보 이름(섹션명 또는 `enum_options[].name`)과 대조하고, 매칭 실패 시 그 항목은 설정 안 함 처리 + 경고 한 줄.
   - 항목이 4개를 넘으면 5번째 이후는 추천값을 그대로 적용하되 6.4 draft에 "(자동)"으로 표시해 Review에서 고칠 수 있게 한다.
   - 물을 항목이 하나도 없으면(섹션 단일 + enum 필드 없음) 이 질문을 생략한다.

5. **GID 변환.** 확정 결과를 프로젝트별로 정리한다. 섹션은 프로젝트별 섹션 GID, 커스텀필드는 해당 프로젝트의 `custom_field_settings`로 `필드명→옵션명`을 `{필드GID: 옵션GID}`로 변환한다(enum이면 옵션명으로 `enum_options[].name`을 찾아 옵션 GID). 겹치는 필드는 양쪽 프로젝트에 같은 값이 들어가되, GID는 각 프로젝트의 설정에서 찾는다(보통 동일 GID).

#### 6.4 Draft 출력

채팅창에 전체 draft를 inline으로 보여준다. 경유 흐름이면 두 프로젝트를 모두 표시한다:

```
Metaboard: 섹션 <섹션명>            ← 경유 흐름일 때만
개별 프로젝트: <프로젝트명> / 섹션 <섹션명>
태스크명: <생성된 태스크명>
담당자: 나(me)
커스텀필드: <필드명=옵션명, ... (자동 적용분은 "(자동)" 표시, 없으면 "없음")>
---
<본문 전체>
---
```

#### 6.5 Review 루프

사용자가 수정 요청을 하면 반영해 draft 전체를 다시 출력한다. 다음까지 반복한다:

- 수락 의사("좋아", "OK", "진행해") → 7단계로
- 중단 의사("취소", "그만") → 조용히 종료

### 7단계: 최종 승인 [절대 건너뛰지 않는다]

`AskUserQuestion`으로 최종 확인을 받는다.

- **question**: `"위 draft로 Asana 태스크를 생성할까요?"`
- **header**: `"태스크 생성 확인"`
- **옵션 1**: `"태스크 생성"` → 8단계로
- **옵션 2**: `"취소"` → 조용히 종료

### 8단계: 태스크 생성

3단계에서 정한 흐름에 따라 갈린다.

#### 개별 흐름 (Metaboard 미경유)

`create_tasks`로 태스크 1개를 개별 프로젝트에 바로 만든다:

- `project_id`: 2단계의 개별 프로젝트 GID
- `section_id`: 6.3의 개별 프로젝트 섹션 GID
- `name`: 6.1의 태스크명
- `html_notes`(또는 `notes`): 6.2의 본문
- `assignee`: `"me"`
- `custom_fields`: 6.3의 매핑 결과를 JSON 문자열로 전달(`create_tasks`의 태스크별 `custom_fields`는 문자열 타입이다). 비어 있으면 생략.

#### 경유 흐름 (Metaboard 먼저, 그다음 개별 연결)

워크플로우 순서대로 두 번의 호출로 만든다. 커스텀필드는 그 필드를 가진 프로젝트에 태스크가 **속한 뒤에야** 설정되므로, 각 필드를 자기 프로젝트 멤버십이 생긴 시점에 넣는다.

1. **Metaboard에 생성** — `create_tasks`:
   - `project_id`: 3단계의 Metaboard GID
   - `section_id`: 6.3의 Metaboard 섹션 GID
   - `name` / `html_notes` / `assignee`: 위와 동일
   - `custom_fields`: 6.3 결과 중 **Metaboard에 존재하는 필드**만 JSON 문자열로. 없으면 생략.
   - 반환된 태스크 GID를 보관한다.

2. **개별 프로젝트에 연결 + 나머지 필드** — `update_tasks`로 위 태스크 GID를 갱신:
   - `add_projects`: `[{project_id: <개별 프로젝트 GID>, section_id: <개별 섹션 GID>}]`
   - `custom_fields`(object 타입): 6.3 결과 중 **개별 프로젝트에만 있는 필드**(Metaboard에서 이미 설정한 것 제외). 같은 호출에서 처리하되, 특정 필드가 "태스크에 없는 필드"류 오류를 내면 그 필드만 빼고 한 번 더 `update_tasks`로 재시도한다(멤버십 반영 타이밍 문제일 수 있음).

**결과 처리(공통):**
- 성공 → 생성된 태스크의 `permalink_url`을 `"태스크 생성됨: <url>"` 형식으로 출력하고 종료. 경유 흐름이면 step 1에서 만든 그 태스크의 링크이며, 연결된 개별 프로젝트도 한 줄로 함께 알린다. 응답에 `permalink_url`이 없으면 보관한 태스크 GID로 `get_task`(`opt_fields: "permalink_url"`)를 호출해 얻는다.
- 실패 → 반환된 오류 메시지를 그대로 보여주고, draft 본문은 이미 화면에 있으므로 다시 시도할 수 있게 안내한다. 경유 흐름에서 1번은 성공했는데 2번에서 실패하면, **Metaboard에는 이미 태스크가 만들어졌음**을 알리고(중복 생성 방지) 2번만 재시도하도록 안내한다.

|---|---|---|
| 1 | Asana MCP 미연결 | 안내 후 종료 |

| 6 | 물을 항목 없음(섹션 단일 + enum 필드 없음) | 6.3 질문 생략, 섹션 추천값·필드 없이 진행 |
